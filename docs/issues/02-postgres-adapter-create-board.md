## Acceptance Criteria

Given an empty `bookmarks` table
When `Create` is called with a valid, non-duplicate URL
Then a row is inserted with `CanonicalURL`/`IdentityHash` derived per
`domain.Canonicalize`/`DeriveIdentityHash`, `Title` defaulted via
`domain.DefaultTitle` if none supplied, `Status = StatusInbox`,
`CreatedAt == UpdatedAt`, and the returned `domain.Bookmark` matches the
inserted row exactly

Given a bookmark already exists with a given `IdentityHash`
When `Create` is called with a URL that canonicalizes to the same hash
Then `Create` returns `*adapter.RepositoryError{Kind: ErrKindDuplicate,
Existing: <the pre-existing row>}` and no second row is inserted — the
unique index on `identity_hash` (ARCHITECTURE_RFC.md "Persistence Schema")
is the race-safety mechanism; catch the unique-violation and re-query
rather than check-then-insert

Given a URL that fails `domain.Canonicalize`'s validation
When `Create` is called
Then `Create` returns `*adapter.RepositoryError{Kind: ErrKindInvalidURL}`
and no row is inserted

Given bookmarks exist across all three statuses with various tags
When `Board` is called with an empty `BoardFilter`
Then all bookmarks are returned grouped by `Status` and ordered by
`Position` within each status

## Out of Scope

`Move`, `Update`, `Delete` (Phase 2/D). Tag filtering in `Board` beyond an
empty filter (Phase D). The exact fractional-rank format for `Position` on
first insert can be the simplest value that satisfies ordering for a
single row (e.g. matching `FakeBookmarkRepository`'s zero-padded-index
convention is NOT required — this is real Postgres, decide the actual
first-insert rank value as part of this issue, since Phase 2 is the first
to need meaningful multi-row rank computation).
